# Agents

## OCR Agent

Performs OCR on PDF files in the `input` folder using the local olmocr-2-7b AI model.

### Setup

1. Ensure the local AI model server is running at `http://127.0.0.1:1234`.
2. Install Python dependencies:
   ```bash
   python3 -m pip install --user requests
   ```
3. Install Poppler (for `pdftoppm`):
   ```bash
   brew install poppler  # macOS
   # or apt-get install poppler-utils on Linux
   ```
4. Install ImageMagick (optional, for resizing):
   ```bash
   brew install imagemagick
   ```

### Usage

Run the agent via Kilo:

```bash
kilo agent ocr
```

This will process all PDF files in the `input` folder and output markdown files in `ocr-output`.

To process a single PDF:

```bash
kilo agent ocr -- --single input/example.pdf
```

### Parameters

- `--input`: Input directory (default: `input`)
- `--output`: Output directory (default: `ocr-output`)
- `--single`: Process a single PDF file
- `--dpi`: DPI for image conversion (default: 150)
 - `--timeout`: Timeout for API requests in seconds (default: 500)
- `--max-retries`: Maximum retries for API requests (default: 3)
- `--force`: Overwrite existing output files

### Notes

- The agent skips PDFs that already have a corresponding markdown file in the output directory.
- If OCR fails for a page, the page will be empty in the output.
- Large PDFs may take time; consider adjusting DPI for speed vs accuracy.

## Parse Agent

Extracts key information from OCR-generated markdown files.

### Setup

No additional setup required beyond the OCR Agent setup.

### Usage

Run the agent via Kilo:

```bash
kilo agent parse
```

This will parse all markdown files in the `ocr-output` folder and display extracted information.

To parse a single markdown file:

```bash
kilo agent parse -- --single ocr-output/example.md
```

### Parameters

- `--input`: Input directory containing markdown files (default: `ocr-output`)
- `--single`: Parse a single markdown file

### Extracted Information

- **Date of Notarization**: The date the document was notarized (extracted from notarial certificate or waiver date field)
- **Document Number**: The notarial document number (Doc. No.)
- **Document Type**: Type of legal document (e.g., Affidavit of Loss, Authorization and Designation, Waiver of Electronic Transmittal, etc.)
- **Page Number**: The page number in the notarial register (Page No.)
- **Book Number**: The book number in the notarial register (Book No.)
- **Series Year**: The series year of the notarial register (Series of)
- **Lastname**: Last name of the primary party (affiant/signer). For corporations/juridical entities, the lastname of the signing party. For waiver documents, extracts lastname from signatory information.

### Notes

- Extraction uses heuristic patterns and may not be 100% accurate.
- Missing fields are reported as "Not found".
- The agent only reads markdown files generated by the OCR Agent.

## Rename Agent

Renames PDF files based on OCR-extracted information using document-type specific formats with ISO standard dates.

### Setup

No additional setup required beyond OCR and Parse agents.

### Usage

Run the agent via Kilo:

```bash
kilo agent rename
```

This will rename all PDF files in the `input` folder and copy them to `renamed/` directory with descriptive names.

To rename a single PDF:

```bash
kilo agent rename -- --single img-260309095401
```

### Parameters

- `--input`: Input directory containing PDF files (default: `input`)
- `--ocr-output`: Directory containing OCR markdown files (default: `ocr-output`)
- `--output`: Output directory for renamed files (default: `renamed`)
- `--single`: Process a single PDF file (without .pdf extension)
- `--template`: Custom filename template using placeholders:
  - `{Lastname}`: Last name of party
  - `{DocumentType}`: Type of document
  - `{Date}`: ISO formatted date (YYYY-MM-DD)
  - `{DocNo}`: Document number
  - `{OriginalName}`: Original filename without extension
- `--dry-run`: Preview changes without renaming files
- `--interactive`: Prompt for confirmation before each rename
- `--force`: Overwrite existing files (use with caution)
- `--skip-existing`: Skip files that already exist in output directory

### Filename Generation Priority

If no template is provided, filenames are generated using document-type specific formats:

#### For notarized documents:
`{Date}-D{DocumentNumber}-{DocumentType}-{Lastname}.pdf`

#### For waiver documents (WAIVER OF ELECTRONIC TRANSMITTAL):
`{Date}-{Lastname}-{DocumentType}.pdf`

#### Fallback logic:
If required fields are missing, the agent uses available fields with this priority:
1. With all required fields (as above)
2. With date, lastname, document type (omitting missing fields)
3. With lastname and document type
4. Original name with document type and '_renamed' suffix
5. Original name with '_renamed' suffix

### Date Format Conversion

The agent converts dates to ISO standard (YYYY-MM-DD):
- "02 FEB 2026" → "2026-02-02"
- "FEB 02 2026" → "2026-02-02"
- "Feb. 02, 2024" → "2024-02-02"

### Notes

- Files are copied (not moved) from `input/` to `renamed/` directory.
- Original files in `input/` remain unchanged.
- Duplicate filenames are handled with numerical suffixes (`_1`, `_2`, etc.).
- The OCR agent must have processed the PDFs first.

## Database Agent

Administers the SQLite database for notary processing center. All other agents delegate to this database for data persistence.

### Setup

No additional setup required beyond other agents. The database is automatically created when first initialized.

### Usage

Run the agent via Kilo:

```bash
kilo agent database
```

Available subcommands:

- `init`: Initialize database with schema
- `import`: Import existing processed documents
- `stats`: Show database statistics
- `backup`: Create database backup
- `query`: Execute custom SQL query (read-only)
- `repair`: Repair database issues

### Examples

```bash
# Initialize database with schema
kilo agent database -- init --force

# Import existing documents
kilo agent database -- import

# Show statistics
kilo agent database -- stats

# Backup database
kilo agent database -- backup --compress

# Query for documents with OCR
kilo agent database -- query "SELECT d.original_filename, COUNT(ocr.id) as page_count FROM documents d LEFT JOIN ocr_results ocr ON d.id = ocr.document_id GROUP BY d.id"
```

### Parameters

Global options:
- `--database`: Path to SQLite database file (default: `notary_processing.db`)
- `--verbose`: Enable verbose logging
- `--no-db`: Disable database logging (for other agents)

### Database Schema

The SQLite database includes 7 main tables:
1. `documents` - Master list of PDF files
2. `ocr_results` - OCR processing results per page
3. `extracted_data` - Parsed document information
4. `rename_operations` - File rename history
5. `processing_logs` - Detailed processing history
6. `error_logs` - Error tracking
7. `audit_log` - Audit trail of all significant actions

### Integration

All other agents (OCR, Parse, Rename) automatically delegate to the database when available. Database logging can be disabled with `--no-db` flag on any agent.

### Notes

- Database is automatically backed up before destructive operations
- SQLite provides embedded, single-file database with no server required
- Full audit trail tracks all processing steps for compliance
- Database can be queried for reporting and statistics
